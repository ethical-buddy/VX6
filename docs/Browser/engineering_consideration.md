## 1. Rendering & Layout

Rendering is the hardest part of any browser, and even if we embed WebKit/Blink we still need to understand what we are actually shipping. We have to account for Webview quirks per OS — WebKitGTK on Linux, WKWebView on macOS, and Edge WebView2 on Windows all behave slightly differently, so CSS rendering, font rendering, scroll behavior, and JS engine versions will not be identical; we cannot assume pixel-perfect consistency across platforms. On Windows, WebView2 requires a runtime — Edge WebView2 must be installed on the user's machine, and if it is missing the app will not render, so we either bundle the fixed-version WebView2 runtime or handle the "not installed" case gracefully with a download prompt. On Linux, WebKitGTK has version fragmentation — Ubuntu 20.04 ships a very different WebKitGTK version than Arch, so we must test on multiple distros. Font rendering is OS-specific too, so we should not hardcode font names and instead use system font stacks and test on each OS.

## 2. Navigation Model

These are the basics every browser needs, and they are easy to overlook if we are not careful. We need a proper history stack for back/forward, not just a URL string, and each tab needs its own stack. We also need session restore so if the app crashes or is closed the user can reopen with their previous tabs. The navigation lifecycle has to cover page load started, DOM ready, page fully loaded, navigation error, and navigation blocked (by the VX6 protocol handler), and each state needs a UI signal like a spinner or error page. Redirect chains matter too — vx6:// resolves to http://127.0.0.1:PORT which may itself redirect, so our protocol handler must not break redirect chains. We also have to handle cancelled navigation when the user types a new URL before the current one finishes and cancel the in-flight request cleanly.

## 3. Security Model

This is the most critical area and also the easiest to get wrong, so we need to be explicit. Same-origin policy has to be defined for our custom vx6:// scheme so two different VX6 services cannot read each other's cookies/localStorage; vx6://alice.ssh and vx6://bob.api must be separate origins. Mixed content needs a decision upfront — if a vx6:// page loads resources from http:// (not HTTPS), we must decide whether to block, warn, or allow; we cannot leave it undefined. We should decide whether we enforce Content Security Policy on VX6-served pages, and the likely answer is yes since we are proxying through localhost. The localhost trust boundary matters because forwarded ports on 127.0.0.1 are accessible to any process on the machine; a malicious local process could try to intercept VX6 sessions, so we mitigate by binding to random high ports, not predictable ones, and rotating them per session. Protocol handler injection is another risk — if a regular web page can trigger vx6:// navigations, that is a security hole, so our custom protocol should only be triggerable from within the VX6 browser itself, not arbitrary web content. Certificate handling needs clarity too: when forwarding a VX6 service over http://127.0.0.1, the webview will not show HTTPS, which is fine because VX6 handles transport encryption below, but we must ensure the user understands this and we do not accidentally show a confusing "not secure" warning. Hidden service security is non-negotiable — invite secrets must never be stored in plaintext, never appear in browser history URLs, and never be logged.

## 4. Process Model

Real browsers use multi-process architecture for isolation and stability, so we need to pick our model with eyes open. A single process is the simplest to build, but if the webview crashes the whole app crashes; that is acceptable for a v1 VX6 browser. One process per tab is much more work, but a tab crash does not kill other tabs, and it is required if we want "production grade." The VX6 runtime as a separate process is non-negotiable — the vx6 node Go process must run independently of the UI process. If the UI crashes, the node keeps running; if the node crashes, the UI should detect it and show a reconnect state, not crash itself. We also need a crash recovery plan for when the VX6 node process dies unexpectedly: the browser must detect it (watch the process or ping the control socket), show an error state, and offer a restart button.

## 5. Tab Management

Each tab needs its own webview instance (or a shared webview with navigation swapping — simpler but worse UX). Each tab needs its own navigation history. Each tab that connects to a VX6 service holds an active forwarded port, and when the tab closes we must release that port and tell the VX6 runtime to stop forwarding. If we do not clean up forwarded ports on tab close, we will exhaust available ports over time, so resource leaks have to be handled.

## 6. Permissions Model

Browsers gate access to camera, microphone, location, notifications, and more, so we need to decide our rules. Do VX6 services get access to device APIs? This probably needs explicit user permission per service and should be stored per VX6 service identity, not per localhost port since ports rotate. File system access is another area — if a VX6 service requests file upload/download, the webview triggers the OS file picker, which is fine, but we must make sure the downloaded file destination is sane. Clipboard access also needs a decision; most browsers ask, and we should too.

## 7. Storage & State

For cookies we must scope to the VX6 service identity (alice.ssh), not the localhost port (which changes), otherwise cookies break every reconnect. localStorage / IndexedDB has the same issue and must be scoped to service identity. Cache is still just HTTP cache for resources served by VX6 services, and we need a max size and eviction policy. Bookmarks should store vx6://service-name, not http://127.0.0.1:PORT. History should do the same and store the VX6 URL, not the forwarded localhost URL, and we must never store invite secrets or hidden service resolution details in history. If multiple users share a machine, their VX6 identities and browser profiles need full separation.

## 8. Networking

Our proxy model should be clear: regular http:// and https:// URLs go through the OS/system network stack as normal, and only vx6:// URLs go through our runtime bridge — we should not accidentally route all traffic through VX6. DNS for regular URLs is handled by the webview and we should not interfere. VX6 is IPv6-native, while our forwarded ports on localhost are 127.0.0.1 (IPv4 loopback); that is fine because localhost forwarding is local and IPv6 is used for actual peer transport, but we must ensure we are not accidentally forcing IPv4 for VX6 peer connections. Timeout handling is also needed — if a VX6 service never becomes reachable, we should set a connection timeout (e.g., 15 seconds) and show an error page rather than a forever-spinning tab. We also need an offline state: if the VX6 node loses all peers, the browser should detect this and surface it clearly.

## 9. UI/UX Requirements

These are easy to underestimate, so we should call them out explicitly. Address bar behavior matters: pressing Enter on a bare service name like alice.ssh should resolve as vx6://alice.ssh, pressing Enter on https://example.com should load normally, and entering garbage should offer a search or show an error. Loading states need to be two-phase — first the VX6 resolution (DHT lookup + port forwarding setup), then the actual page load, with distinct UI states so the user knows if the delay is VX6 or the service itself. Error pages have to be custom for service not found in DHT, service found but unreachable, connection timed out, hidden service invite required, and signature verification failed. Node status must always be visible so the user can see whether the VX6 node is running, how many peers are connected, and whether DHT is healthy; we should not hide this in a buried settings screen. A tray icon is also needed on all platforms — since the VX6 node runs as a background process, a system tray icon (Linux, macOS, Windows) lets the user know it is running and gives a quick way to open the browser or stop the node.

## 10. Cross-Platform Packaging Requirements

We also need to be realistic about packaging. Code signing is required — macOS needs app signing and notarization or users get a "cannot be opened" Gatekeeper warning, which means an Apple Developer account ($99/year). Windows SmartScreen will flag unsigned executables, so we need a code signing certificate. Auto-update should be built in from day one because users will not update manually; Tauri has tauri-plugin-updater for this, and we should define the update server endpoint early. The bundled binary per platform is non-negotiable — the vx6 Go binary must be compiled for linux/amd64, linux/arm64, darwin/amd64, darwin/arm64 (Apple Silicon), and windows/amd64, which is five separate binaries to ship and keep in sync with the browser version. Installer behavior needs to handle first install, upgrade, uninstall, and the case where WebView2 runtime is missing on Windows; on macOS drag-to-Applications must work; on Linux both .deb and .AppImage are expected. Finally, startup on login is required: systemd user unit on Linux, launchd plist on macOS, and a registry run key on Windows, each with its own quirks.
